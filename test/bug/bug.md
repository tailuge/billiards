## Additional Groundwork

### Browser replay findings

Added temporary replay logging in `src/controller/replay.ts` and ran a local browser build with the replay URL above.

Observed at shot 1 start:

```json
{
  "shotType": "AIM",
  "currentCueball": {
    "x": -0.7204999923706055,
    "y": 0.44873636960983276
  },
  "recordedAimPos": {
    "x": -0.7204999923706055,
    "y": 0.44873636960983276
  },
  "distance": 0
}
```

Observed after shot 1 settles, before shot 2 is applied:

```json
{
  "cueball": {
    "x": 0.6128004193305969,
    "y": 0.028110530227422714
  },
  "remainingShots": 20
}
```

Shot 2 recorded `aim.pos` is:

```json
{
  "x": 0.6128116250038147,
  "y": 0.02814456634223461
}
```

So the replay-side mismatch at the shot boundary is approximately:

- `dx ~= 0.0000112056732178`
- `dy ~= 0.0000340361148119`
- `d ~= 0.0000358332837422`

This is a strong signal that the break replay ends in a cueball position that is extremely close to, but not identical to, the recorded start position of shot 2.

### Important conclusion

This makes a broad “rogue input after hit” explanation less likely.

The most useful current working theory is:

1. shot 1 replay physics ends at a slightly different settled state than the original live shot 1
2. the break amplifies that tiny difference, so several balls visibly disagree from shot 2 onward

This still points more to a shot-boundary mismatch than to random instability.

### Test harness findings

A regression test was added at `test/bug/break_replay_regression.spec.ts`.

That test can reproduce a stable mismatch in the local Jest harness, but the harness currently lands at a much larger delta than the browser:

- direct fixed-step path: `0.03903063840011522`
- browser-like `1/60` frame loop in Jest: also `0.03903063840011522`

The browser build does **not** show this larger discrepancy. Therefore the Jest harness is currently not a faithful reproduction of the exact runtime path/state used by the local browser replay for this bug.

Do not treat the current Jest magnitude as evidence about the real replay bug. The useful part of the test is that it provides a deterministic local repro point for future investigation.

### Logging recommendation

After the experiment, the replay logging was reduced to a minimal form:

```ts
console.log("replay-shot-delta", {
  dx: aim.pos.x - this.container.table.cueball.pos.x,
  dy: aim.pos.y - this.container.table.cueball.pos.y,
  d: this.container.table.cueball.pos.distanceTo(aim.pos),
})
```

This is worth keeping while investigating because:

- it is concise
- it survives replay-only investigation
- `dx` and `dy` are more informative than only logging `distance`

### Stronger leads, weaker leads

Stronger current leads:

- `aim.pos` at the next shot boundary is not identical to the settled cueball position
- the discrepancy appears immediately after the break, which narrows the area a lot
- break physics is collision-sensitive, so a tiny boundary mismatch can fan out into larger visible differences

Weaker leads:

- random nondeterminism in `table.ts`
- rogue user input after hit
- `serialiseHit()` only sending ball 0 for spectator sync

That last point matters for watch/spectate paths, but the replay URL assembled by `Recorder.wholeGame()` is driven by `entries[0].state` plus recorded shot events, not by `HitEvent.tablejson.balls`.

### Best next steps if revisiting

If returning to this later, the best focused question is:

> Should `recorded AIM.pos for shot N` always equal `settled cueball.pos immediately before shot N`?

If temporary debug data is added again, the most useful low-cost thing to record per shot is **explicit cueball x/y at shot start** alongside the `AimEvent`, rather than serialising all moved balls.

That would directly test whether `aim.pos` is the wrong source of truth for replay shot starts.
